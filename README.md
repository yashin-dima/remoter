# Remoter

Mac-приложение для работы с проектами: дерево файлов, наглядный git-diff (две колонки,
зелёное/красное, подсветка синтаксиса), редактирование и терминал — всё в одном окне.
Проект может лежать **на сервере** (по ssh) или **на этом же Mac** — приложение работает
с обоими одинаково.

Задумано под один сценарий: **Claude Code работает с кодом на сервере, а вы видите, что именно
он меняет** — прямо в diff'е, который обновляется сам. Claude при этом запущен **у вас на Mac**,
а к проекту ходит по ssh.

**На сервер ничего устанавливать не нужно.** Всё делается через обычный `ssh` и `git`.

## Быстрый старт

```sh
./build.sh                       # собрать Remoter.app (Monaco подтянется сам)
open Remoter.app
```

Для сборки нужны **Xcode 15+** (или Command Line Tools со Swift 5.9+) и **Node.js с npm** —
последний только чтобы один раз подтянуть Monaco с npm-реестра при первой сборке (нужна сеть).
Дальше Monaco лежит локально и сеть больше не требуется.

Необязательный шаг перед первой сборкой:

```sh
./Scripts/make-signing-cert.sh   # стабильная подпись, чтобы macOS не переспрашивала разрешения
```

Без него `build.sh` подпишет приложение adhoc — всё работает, но при каждой пересборке macOS
считает приложение новым и заново спрашивает доступ (к связке ключей, уведомлениям). Скрипт
кладёт самоподписанный сертификат в вашу связку ключей `login` и делает его доверенным для
подписи кода — macOS спросит на это пароль. Это разовая локальная операция, наружу ничего
не уходит.

Дальше «Добавить проект» → сервер, потом **«Обзор…»**: путь выбирается кликами прямо на сервере,
помнить его наизусть не нужно. Папки с git-репозиторием в обзоре помечены.

Сервер — это то же, что вы пишете в `ssh`: алиас из `~/.ssh/config`, `user@example.com`, что угодно.
Весь ваш `~/.ssh/config` работает как есть; сам файл приложение не читает и не правит.

### Проект на этом же Mac

Проект не обязан лежать на сервере. При добавлении выбирается **«На этом Mac»** — и дальше обычная
папка: ssh не нужен, git может отсутствовать вовсе (тогда просто не будет раздела с изменениями).
Всё остальное работает ровно так же: дерево, diff, редактор, терминал, Claude во вкладках.

Устроено это без единой ветки «если локальный» в логике: и дерево, и git, и запись файла общаются
с проектом одним способом — отдают POSIX-скрипт в канал. У серверного проекта скрипт исполняет ssh,
у локального — `/bin/sh` прямо здесь. Разница живёт в одном классе (`Connection`), а не размазана
по приложению.

**Чужая папка остаётся чужой.** В локальный проект приложение не пишет ни `CLAUDE.md`, ни скрипт
`remote`, ни `docs/` — у проекта уже может быть свой `CLAUDE.md` (у самого Remoter он есть), и
заводить в чьём-то репозитории свои файлы недопустимо. Claude запускается прямо в папке проекта,
а единственное, что там появляется, — хуки уведомлений в `.claude/settings.local.json` (машинный
файл самого Claude Code, чужие настройки в нём не трогаются). Скриншоты из буфера тоже уходят не
в репозиторий, а в служебный каталог приложения — чтобы не пачкать вам `git status`.

Проектам можно указать **компанию** — список тогда разбивается на разделы. Сверху **поиск**: он
идёт сразу по всем проектам, и не только по названию, но и по компании, серверу и пути.

Открыть проект сразу, минуя список: `open -a Remoter --args --open <название>`.

## Что в окне

**Слева — три раздела:**

| | |
|---|---|
| **Git** | что тронуто в проекте, с числом в бейдже; клик открывает diff |
| **Remote** | дерево проекта (у проекта на этом Mac раздел так и называется — **Файлы**) |
| **Local** | папка проекта на вашем Mac: документация, заметки, инструкция для Claude, конфиг доступа. У локального проекта раздела нет: рабочая папка Claude — сам проект |

**Сверху — вкладки:** сначала разговоры с Claude, потом терминалы, потом файлы. Всё, кроме файлов,
живёт своим процессом и не умирает от переключения вкладки.

**Справа вверху — кнопки:** «**+**» открывает ещё проект, рядом — **терминал** (`⌥⌘T`: на сервере
или в папке проекта — смотря какой проект),
дальше обновление и часы с прошлыми разговорами. Отдельного раздела «Terminal» в колонке больше нет:
он ничего не показывал, а только открывал — а открытые терминалы и так стоят вкладками сверху.

**Проектов можно открыть сколько угодно** — по окну на каждый. У окна свой канал, своя папка
и свои сессии Claude: проекты друг о друге не знают и друг другу не мешают, даже если сервер
у них один. Список проектов закрывается, как только проект открыт, и **сам больше не всплывает
никогда** — позвать его обратно можно только кнопкой «**+**» в правом верхнем углу (или `⇧⌘O`).

## Claude: запущен локально, работает по ssh

**Claude запускается на вашем Mac** — ваши токены, ваша сессия. А с файлами проекта он работает
**по ssh**, ровно так же, как это делают руками в терминале. (У проекта на этом Mac никакого ssh
нет вовсе: Claude запускается прямо в папке проекта, и всё, что ниже про `~/Remoter/<проект>`,
к нему не относится.)

**Сервер про Claude не знает ничего** — и не должен. Ни инструкции, ни скриптов, ни следов: всё
это живёт в локальной папке `~/Remoter/<проект>/`, которую видно в разделе **Local**:

```
CLAUDE.md    где сервер, где каталог проекта, как с ним работать
             (Claude читает это при каждом запуске)
remote       короткая обёртка над ssh:  ./remote git status
                                        ./remote cat src/main.py
                                        ./remote grep -rn "TODO" src/
docs/        ваша документация и заметки
```

Под `remote` — обычный ssh, просто через **уже открытый канал** Remoter: без рукопожатия, без
пароля, за десятки миллисекунд. Хост и путь заодно лежат в `$REMOTER_HOST` и `$REMOTER_PATH`,
так что можно звать `ssh` и напрямую.

**CLAUDE.md можно дописывать.** Приложение обновляет в нём только свой блок — тот, что между
`<!-- remoter:begin -->` и `<!-- remoter:end -->` (сервер, каталог, как ходить). Всё, что вы
написали вне блока, переживает и переоткрытие проекта, и смену хоста: раньше файл переписывался
целиком, и заметки для агента пропадали при каждом запуске.

**Одинаковые имена не сталкиваются.** Папка помечена id проекта (`.remoter-id`), и второй проект
с тем же именем — на другом сервере или у другой компании — получает свою: `имя (хост)`. Иначе два
«api» делили бы один каталог, и Claude второго ходил бы по ssh-каналу первого.

**Никаких локальных копий и синхронизаций.** Claude читает и правит файлы прямо на сервере —
поэтому его правка появляется в вашем diff'е сразу, а не после отдельного шага.

### Несколько разговоров сразу

Над проектом идёт несколько задач, и открывать новый разговор, обрывая текущий, бессмысленно.
Поэтому **каждая сессия — своя вкладка, свой терминал и свой процесс**. Кнопка «**+**» в ряду
вкладок открывает новую **рядом** с работающей, не трогая её.

Перед запуском спрашиваются параметры: модель, уровень reasoning, режим работы. Собранная команда
показана целиком, до запуска.

**По умолчанию приложение не навязывает Claude ничего.** Модель и effort он берёт свои — те, что
стоят у него в `~/.claude/settings.json` (их ставят `/model` и `/effort`). Это его настройки,
а не наши, и подменять их молча приложение не имеет права: подставляя `--model opus` поверх вашего
`opus[1m]`, оно обрезало контекст с миллиона до двухсот тысяч, ничего об этом не сообщая.

Выбрать модель явно можно — тогда в команду уйдёт `--model 'opus[1m]'`, **в кавычках**: `[1m]` —
валидный алиас, но для шелла квадратные скобки это шаблон имени файла, и без кавычек zsh отвечает
`no matches found: opus[1m]`, а Claude не запускается вовсе.

### Плашка сессии

Показывает, **что у сессии происходит сейчас**: заполненность контекста, модель, effort, режим
работы. Не флаги запуска, а текущее состояние — набрали внутри `/effort xhigh` или переключили режим
на shift+tab, и плашка переедет следом (она читает журнал, который Claude ведёт сам).

**Кольцо слева — контекст.** Он кончается внезапно и всегда некстати: Claude уходит на авто-сжатие
посреди задачи и теряет половину того, что вы ему объясняли. Видя, что окно подходит к концу, можно
сжать его самому (`/compact`) — в удобный момент, а не в случайный. Жёлтое с 60%, красное с 85%;
в подсказке — сколько токенов из скольких.

Занятое считается по последнему ответу Claude (его вход вместе с прочитанным из кэша плюс выход —
это и есть то, что занимает окно). Реплики подагентов (`isSidechain`) в счёт не идут: у них свой
контекст, и кольцо от них прыгало бы на чужие цифры.

А вот **размер** окна числом не отдаёт никто: в журнал Claude его не пишет, а `/context` рисует
картинку в терминале и наружу ничего не возвращает. Поэтому он берётся из алиаса, с которым сессия
работает (`[1m]` → миллион, иначе 200k), — причём из **текущего**, а не из того, с которым её
запустили: набрали внутри `/model sonnet` — окно стало 200k, и те же 120k занятого превращаются
из 12% в 60%. Если же занято окажется больше предполагаемого, кольцо само поправится, а не станет
врать про «переполнено».

**Плашки кликаются.** Выбор модели, effort или режима **уходит в терминал командой** — той же,
что набрали бы руками (`/model opus`, `/effort xhigh`, shift+tab для режима). Никакой скрытой
подмены состояния: команду видно, её можно прервать, а плашка переключится, только когда смена
реально произойдёт. Не бывает так, что в приложении одно, а у Claude другое.

**«Стоп»** появляется, когда Claude работает, и прерывает текущий ответ (Escape). Сессия при этом
жива — это не Ctrl+C, который, нажатый дважды, из Claude выходит. Что сессия занята, приложение
узнаёт от его же хуков; поэтому и во вкладке видно, которая из них сейчас думает.

Крестик на вкладке закрывает сессию. Разговор при этом не пропадает — он записан в журнал Claude
и открывается заново из списка сессий.

### Сессии

Часы в тулбаре (`⇧⌘J`) показывают прошлые разговоры по этому проекту: заголовок, когда это было,
сколько сообщений. «Продолжить» открывает любой из них — **отдельной вкладкой**, не занимая ту,
где вы работаете. В терминал уходит обычный `claude --resume <id>`, видимый целиком.

Журнал ведёт **сам Claude Code** (`~/.claude/projects/…`), приложение его только читает и никогда
не пишет — это проверяется тестом. Заголовок разговора Claude придумывает не сразу; пока его нет,
в списке стоит первая фраза, с которой всё началось.

**При открытии проекта продолжается последний разговор.** Работа идёт неделями и не заканчивается
вместе с окном: вернувшись, вы оказываетесь там же, где остановились.

### Файлы и скриншоты в запрос

Три способа, все дают одно и то же — путь в строке ввода Claude (он умеет читать картинки по пути):

- **перетащить** файл в терминал;
- **⌘V** со скриншотом в буфере — картинка сохранится в `<проект>/.attachments/` и путь подставится;
- **скрепка** в плашке над терминалом.

`Enter` при этом не нажимается: подставить путь и отправить запрос — разные решения.

### Уведомления

Когда Claude заканчивает работу или просит разрешение, приходит уведомление macOS: проект,
название сессии и последняя реплика — даже если окно приложения прямо перед вами. Клик открывает
**то самое окно и ту самую вкладку**, откуда уведомление пришло.

Работает это через хуки самого Claude Code (`Stop` и `Notification`), которые приложение ставит
в `~/Remoter/<проект>/.claude/` при открытии проекта. Чужие настройки и чужие хуки в этом файле
не трогаются.

Хук стучится прямо в локальный http-сервер приложения (тот же, на котором живёт редактор), а не
через `open remoter://…`, как было сначала: `open` идёт через Launch Services, а те на каждую
доставку «переоткрывают» приложение — и оно распахивало окно со списком проектов поверх работы.

**Звук** — свой на каждое событие (настройки, ⌘,): системный из списка macOS или ваш файл. Разные
звуки стоит поставить не для красоты: «он ждёт меня» слышно, не отрываясь от другого окна. Свой файл
копируется в `~/Library/Sounds` — иначе macOS его просто не найдёт и молча сыграет стандартный.

Не приходят вовсе — Системные настройки → Уведомления → Remoter. Проверить всю цепочку:
`./Remoter.app/Contents/MacOS/Remoter --selfcheck`.

## Настройки (⌘,)

Они у каждого свои, поэтому живут в отдельном окне, а не зашиты в код:

- **Масштаб интерфейса**, 80–160%. Меняются и шрифты, и высота строк, и зоны нажатия — целиком,
  а не только текст: увеличь один шрифт, и он перестал бы помещаться в кнопки.
- **Шрифт терминала и редактора** — отдельно от масштаба: в них смотрят часами, и там свой
  комфортный размер.
- **Звук уведомлений** — свой для «закончил» и для «спрашивает».
- **С чем открывать Claude** — модель, effort, режим, контекст 1M, продолжать ли последний разговор.
- **Предпросмотр файла** по одиночному клику — можно выключить.

## Git

Панель сделана по логике расширения git в VS Code, и не ради подражания. Прежняя врала в главном:
файл, изменённый **и** в индексе, **и** в рабочей копии (`MM`), попадал только в раздел «В индексе»,
а его несохранённая правка нигде не показывалась — коммит уносил не то, что было видно на экране.

Теперь такой файл честно стоит в обоих разделах, с разными буквами: в индексе он, например, `A`,
а в изменениях `M`. Это два разных изменения, и делать с ними надо разное.

- Сверху — **ветка** и насколько она разошлась с сервером (↑ не отправлено, ↓ не забрано).
- Дальше — **поле коммита** прямо в панели. На кнопке написано, что она сделает: «Закоммитить
  индекс (3)» или «Закоммитить всё (7)», если индекс пуст. Никакой лотереи.
- Разделы: **Конфликты**, **В индексе**, **Изменения** — с числом и кнопками на весь раздел
  (добавить всё, убрать всё, отбросить всё).
- В строке — кнопки на сам файл: `+` в индекс, `−` из индекса, `↩︎` отбросить. Отбрасывание
  спрашивает, и не «вы уверены?», а сколько файлов и что именно пропадёт.

## Файлы

**Одиночный клик — предпросмотр** (как в VS Code): вкладка открывается курсивом и будет заменена
следующим таким же кликом. Двойной клик оставляет файл в ряду насовсем; первая же правка — тоже.
Иначе, прощёлкивая diff, за минуту набираешь два десятка вкладок. Фокус при этом сразу уезжает
в редактор: открыл — печатай, а не кликай второй раз, чтобы туда попасть.

Постоянные файлы открываются рядом, а не вместо друг друга. Буква статуса (M, A, ?) и цветная
полоска показывают, что с файлом; точка справа — есть несохранённые правки. Каждая вкладка помнит
свой скролл, курсор и историю отмены.

**Дерево принимает файлы перетаскиванием** и умеет обычные файловые операции — над одним файлом
и над пачкой (⌘-клик, ⇧-клик, ⌘A):

| | |
|---|---|
| `⌘C` / `⌘X` / `⌘V` | копировать / вырезать / вставить |
| `⌘⌫` или `⌫` | удалить |
| `⏎` | переименовать |
| `⎋` | снять выделение |

Всё то же есть в контекстном меню, плюс «Новая папка» и «Копировать путь». Удаление и вставка
происходят на сервере, поэтому удаление спрашивает подтверждение, а копирование и вставка
**никогда не затирают** одноимённый файл — он ляжет рядом под свободным именем. Корень проекта
удалить нельзя в принципе.

## Как это работает

### SSH — системный, а не свой

Приложение не реализует SSH-протокол. Оно запускает системный `/usr/bin/ssh` в режиме
мультиплексирования:

```
ssh -M -S <сокет> -N <хост>          # мастер-соединение, держится в фоне
ssh -S <сокет> <хост> '<команда>'    # каждая команда — по уже открытому каналу
```

Что это даёт:

- Бесплатно работает весь ваш `~/.ssh/config`: алиасы, ключи, `ProxyJump`, ssh-agent, known_hosts.
- Команда выполняется за десятки миллисекунд — рукопожатия нет, канал уже открыт.
- Терминалы подключаются через тот же сокет: открываются мгновенно и не спрашивают пароль второй раз.
- Ничего не нужно ставить на сервер и нечему там «протухнуть».

Сокет свой у **каждого проекта**, а не у каждого сервера: два проекта на одной машине — два канала.
Иначе закрытие одного окна уносило бы сокет из-под второго.

Пароль или парольную фразу, если они нужны, спрашивает нативный диалог
([Scripts/askpass.sh](Scripts/askpass.sh)): у фонового мастер-соединения нет терминала, и
спросить в консоли ему негде.

### Diff — это чистый git

- левая сторона: `git show HEAD:<путь>` (для переименованного файла — по **старому** имени,
  иначе diff показал бы «файл целиком новый»)
- правая сторона: содержимое рабочей копии
- список изменений: `git status --porcelain=v2 -z` — машинный формат, поэтому пробелы, кавычки
  и юникод в именах не ломают разбор

Переключатель «vs HEAD / vs Индекс» меняет только левую сторону.

### Живое обновление

Раз в 2.5 секунды по открытому каналу дёргается `git status`. Если вывод не изменился — ничего
не перерисовывается.

Отдельная тонкость: когда агент правит файл, который git **и так** считает изменённым, вывод
`git status` не меняется — по нему правку не поймать. Поэтому у открытого файла дополнительно
сверяется контрольная сумма (`cksum`, одна строка вместо перекачивания файла). Содержимое
подменяется без пересоздания моделей Monaco, так что позиция скролла и курсор остаются на месте.

На неактивном окне поллинг встаёт на паузу.

### Редактор — Monaco в WKWebView

Diff и подсветку рисует Monaco (движок VS Code). Он раздаётся **с локального http-сервера на
127.0.0.1**, а не с `file://`, и это принципиально: Monaco считает diff в web worker'е, а из
`file://` воркеры блокируются — diff бы просто не появлялся. В пути стоит случайный токен,
чтобы к серверу не могли обратиться другие процессы на машине.

## Сохранность данных

Приложение ходит в живые проекты, в которых прямо сейчас может работать агент. Поэтому набор
изменяющих операций сознательно крошечный, и каждая выполняется **только по явному действию
пользователя**:

| Операция | Чем вызывается |
|---|---|
| запись файла | ⌘S или кнопка «Сохранить» |
| загрузка файла | перетаскивание в дерево |
| копирование / перемещение | ⌘C/⌘X + ⌘V или меню |
| удаление, переименование | ⌘⌫ / ⏎ или меню, с подтверждением |
| `git add` / `git reset` | кнопки в списке изменений |
| `git checkout --` (откат) | пункт меню + подтверждение |
| `git commit` | окно коммита |

**Просмотр не пишет ничего.** Дерево, diff, поллинг, переключение файлов — только чтение
(`git status`, `git show`, `cat`, `ls`, `cksum`). Это закреплено тестом
`testBrowsingAndPollingNeverModifyAnything`: он снимает слепок всех файлов с размерами и правами,
прогоняет по проекту всё, что умеет приложение, и сверяет слепок побайтно.

### Что защищает запись файла

- **Атомарность.** Пишем во временный файл рядом и переставляем через `mv`. Наполовину записанный
  файл никогда не окажется на месте настоящего.
- **Проверка длины.** Оборвись ssh посреди передачи — `cat` на сервере увидит EOF и завершится
  **успешно**, а на месте файла останется обрубок. Поэтому размер сверяется до байта, и `mv`
  происходит, только если доехало всё. Иначе оригинал остаётся нетронутым, а вы видите ошибку.
- **Симлинки.** Голый `mv` заменил бы ссылку обычным файлом. Идём по ссылкам до настоящего файла.
- **Права.** `+x` у скриптов переживает сохранение.
- **Кодировка.** Файл не в UTF-8 (windows-1251 и т.п.) открывается **только на чтение**: обратно
  мы пишем UTF-8, и «сохранение» даже без единой правки перекодировало бы его целиком.
- **Чужие правки.** Перед записью содержимое на сервере сверяется с тем, что мы оттуда прочитали.
  Если файл успел измениться — приложение спросит (перезаписать / перечитать / отменить), а не
  затрёт молча.
- **Ваши несохранённые правки.** Живое обновление никогда не трогает редактор, в котором есть
  несохранённый текст. Переключатель «vs HEAD / vs Индекс» — тоже: с несохранёнными правками
  база diff'а не переключается, пока вы их не сохраните или не отмените.
- **Обрыв связи — это ошибка, а не пустота.** ssh отдаёт свой код (255) на умерший канал, и он
  больше не путается с ответом команды. Раньше оборванное чтение выглядело как «файла нет в
  ревизии» — изменённый файл рисовался целиком зелёным, — а сохранение, не сумев сверить чужие
  правки, писало вслепую. Теперь видно сообщение, правки остаются несохранёнными, ⌘S можно
  повторить, когда связь вернётся.

### Режим «только чтение»

В настройках проекта есть переключатель **«Только чтение»**. В нём приложение не выполняет на
сервере ни одной изменяющей команды — проверка стоит в одной точке на каждую операцию, а не
«кнопки спрятаны». Если сомневаетесь в проекте — включайте: смотреть diff можно, испортить нельзя.

### Чего приложение не гарантирует

Честно, чтобы не было сюрпризов:

- **Терминал — это настоящий shell.** Что вы (или агент) в нём выполните — целиком на вашей
  совести, приложение туда не вмешивается. По умолчанию Claude запускается с полным доступом.
- **Между проверкой «файл не изменился» и записью проходят миллисекунды.** Если ровно в этот
  промежуток файл перепишет кто-то ещё, его правка потеряется. Окно крошечное, но не нулевое.
- **Смешанные переводы строк** в одном файле редактор приведёт к одному виду при сохранении.
- **`git checkout --` и удаление нового файла необратимы** — это сам git, подтверждение спрашивается.

## Разработка

Нужны **Xcode 15+ / Swift 5.9+** и **Node.js с npm** (последний — только для первой загрузки
Monaco, см. ниже). Единственная внешняя зависимость Swift — [SwiftTerm](https://github.com/migueldeicaza/SwiftTerm),
её тянет SwiftPM сам.

```sh
swift build                 # собрать
./build.sh                  # собрать и упаковать в Remoter.app
./Scripts/vendor-monaco.sh  # подтянуть Monaco в Web/vs (build.sh зовёт сам, если его нет)
./Scripts/make-icon.sh <картинка.png>   # пересобрать иконку
```

Monaco (13 МБ чужого минифицированного кода) в репозитории **не лежит** — он ставится скриптом
одной командой (`npm pack monaco-editor` с реестра npm, нужна сеть) и обновляется сменой версии,
а не коммитом.

### Как разложен код

`Sources/Remoter/Core/` — всё, что общается с внешним миром, и состояние окна:

| | |
|---|---|
| `Connection` | канал к проекту: `SSHConnection` (ssh-мультиплексор) или `LocalConnection` (`/bin/sh` здесь же). Единственное место, где проект «на сервере» отличается от «на этом Mac» |
| `Proc` | запуск процессов; сюда же стекаются таймауты, обрывы и SIGPIPE |
| `RemoteFS`, `LocalFS`, `Git` | чтение и запись файлов на сервере и на Mac, разбор `porcelain=v2` |
| `Workspace`, `LocalWorkspace`, `Settings` | список проектов, папка `~/Remoter/<проект>` (у локального проекта её нет — Claude работает в самом проекте), настройки |
| `MonacoServer`, `MonacoBridge` | локальный http-сервер и мост Swift ⇄ JS |
| `ClaudeConfig`, `ClaudeLive`, `ClaudeSessions` | чтение настроек и журналов Claude Code — только чтение |
| `Notifications` | хуки Claude Code и уведомления macOS |

`WorkspaceModel` — модель окна проекта. Обязанностей у неё много, поэтому она разложена по
файлам-по-обязанности, а не свалена в один: `+Tabs` (ряд вкладок), `+Documents` (открытие, diff,
сохранение), `+Tree` (серверное дерево), `+FileOps` (буфер обмена, удаление, загрузка), `+Git`,
`+Claude` (сессии и терминалы), `+LocalPanel`. Хранимые свойства Swift разрешает держать только
в основном файле — там же карта, что в каком расширении лежит. Логика раскрытия дерева общая
(`FileTreeState`): серверное и локальное деревья отличаются только источником детей.

`Sources/Remoter/UI/` — SwiftUI-слой. Цвета, отступы и шрифты — **только** через `Design.swift`
(`Theme` и `D.s(...)`), чтобы масштаб интерфейса менял всё разом, а тёмная тема не разъезжалась.

### Где лежат данные пользователя

| | |
|---|---|
| `~/Library/Application Support/Remoter/workspaces.json` | список проектов (рядом — `.backup.json`) |
| `~/Remoter/<проект>/` | локальная папка проекта: `CLAUDE.md`, `remote`, `docs/` |
| `~/.remoter/sockets/` | сокеты ssh-мастеров (по одному на открытый проект) |

**Ничего из этого нельзя трогать при отладке.** Однажды уборка после тестового прогона стёрла
список проектов — вместе с настоящими. Поэтому все пути переопределяются переменными окружения:
`REMOTER_STORE` (файл списка), `REMOTER_HOME` (корень локальных папок) и `REMOTER_SOCKETS`
(каталог сокетов). `Tests/local-sshd.sh` печатает их, и запускать тесты и отладочные сборки нужно
**только** с ними. Ещё одна, `CLAUDE_CONFIG_DIR`, уводит в сторону журналы Claude Code.

Приложение раньше звалось **SSHDiff**. Список проектов при первом запуске подхватывается из старой
папки (копированием — оригинал остаётся на месте), а вот корень `~/SSHDiff`, если он есть,
остаётся рабочим и не переименовывается: Claude Code хранит журналы разговоров в каталоге, имя
которого собрано **из пути** рабочей папки, и переезд отвязал бы всю историю сессий разом.

### Тесты

Логику проверяют интеграционные тесты против настоящего sshd и настоящего git — мокать тут
нечего, вся суть приложения в стыках с внешними инструментами: разбор `--porcelain=v2 -z`,
кавычки в путях, атомарная запись, разрешение симлинков.

Без стенда больше половины тестов пропускается (и честно об этом пишет) — «зелёный» `swift test`
на голой машине ничего не доказывает. Полный прогон — только с переменными, которые печатает
скрипт ниже.

```sh
./Tests/local-sshd.sh       # поднимет sshd на 127.0.0.1:2222 и тестовый репозиторий,
                            # а потом напечатает готовую команду для swift test
./Tests/local-sshd.sh stop  # убрать за собой
```

sshd поднимается от вашего пользователя на непривилегированном порту: `sudo` не нужен, «Удалённый
вход» в системных настройках включать не надо, и `~/.ssh` скрипт не трогает вообще — у него свой
ключ, свой `known_hosts` и свой `authorized_keys`.

### Проверка веб-половины

```sh
./Remoter.app/Contents/MacOS/Remoter --selfcheck
```

Проверяет, что Monaco нашёлся в бандле, отдался с локального сервера, посчитал diff и нарисовал
зелёные и красные строки. Это **нельзя** проверить обычным XCTest: WebKit считает страницу
скрытой в любом окне, не принадлежащем нормальному GUI-приложению, и замораживает отрисовку —
в тестовом процессе Monaco не рисует ни строчки.

Стоит прогонять после каждого обновления Monaco. Сломавшийся редактор выглядит совершенно
рабочим — подсветка синтаксиса на месте, — просто зелёного и красного не появляется никогда.
Ровно так и проявилась главная ошибка при разработке: Monaco выводил адрес воркера от корня
сервера, без токена, получал 404 и молча ронял воркер, который считает diff.

Отладка сети: `REMOTER_DEBUG_HTTP=1` — печатает каждый запрос к локальному серверу.
`REMOTER_WEB_ROOT=<путь>` — брать `Web/` не из бандла, а из исходников (правки в `editor.js`
видны после перезапуска, без пересборки).

## Ограничения

- Логин-шелл на сервере должен быть POSIX-совместимым (`sh`, `bash`, `zsh`, `ksh`). С `fish`
  как логин-шеллом экранирование команд поедет.
- Файлы больше 4 МБ и бинарные не открываются — показывается заглушка.
- macOS 14+.
